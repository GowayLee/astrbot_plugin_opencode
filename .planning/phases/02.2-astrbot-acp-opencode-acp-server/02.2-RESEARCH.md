## Key Findings

### 1. AstrBot 官方资料足够支撑“开发参考手册”整理

- 官方插件开发指南至少包含 10 篇可直接引用的文档：`simple.md`、`plugin-config.md`、`session-control.md`、`send-message.md`、`listen-message-event.md`、`ai.md`、`env.md`、`storage.md`、`html-to-pic.md`、`other.md`。
- 对本仓库最相关的主题不是“所有 API 全抄”，而是：插件入口与注册、`main.py` 约束、`_conf_schema.json` 配置模型、消息发送、会话控制、存储与 AI/平台能力接入。
- 因为 D-01 / D-12 要求“整理稿不替代原文”，仓库文档应采用“适用场景 + 关键机制 + 本仓库最相关点 + 官方链接”结构，而不是复制整篇教程。

### 2. ACP 协议文档天然适合按生命周期整理

- `overview` 给出了最稳定的总框架：`initialize` → `session/new|load` → `session/prompt` → `session/update` / `session/request_permission` / `session/cancel`。
- `session-setup` 明确了几个后续实现常用硬约束：路径必须是绝对路径、`cwd` 是 session 级边界、`loadSession` 能力必须先协商、MCP server 至少要支持 stdio。
- `tool-calls` 对 Phase 3/4 尤其关键，因为它定义了 `tool_call`、`tool_call_update`、permission request、状态枚举、diff/terminal/content 表达方式。
- `llms.txt` 说明除上述页面外，还应该在文档中补充索引：`content`、`slash-commands`、`session-modes`、`agent-plan`、`schema`、`file-system`、`terminals`、`initialization`。

### 3. OpenCode ACP 文档量不大，但需要明确“支持范围 + 与 ACP 的关系”

- OpenCode 官方 ACP 页面核心信息集中：通过 `opencode acp` 以 stdio 子进程方式提供 ACP Agent。
- 文档已经明确：OpenCode 在 ACP 下支持内置工具、自定义工具/斜杠命令、MCP server、`AGENTS.md` 项目规则、formatters/linters、agents 与 permissions。
- 当前已知限制是部分内置 slash commands（如 `/undo`、`/redo`）暂不支持。
- 因为 OpenCode 文档本身更偏“如何在编辑器里接入”，仓库整理稿应主动把它映射为“对本项目意味着什么”：本仓库使用本地 stdio ACP 链路，不做远程传输扩展；OpenCode 作为 ACP server，AstrBot 插件作为 client 侧宿主集成。

## Planning Implications

### Recommended Artifact Set

按 D-06 / D-07 / D-08，建议最小但完整的产物集为：

1. `docs/references/astrbot/README.md`
2. `docs/references/acp/README.md`
3. `docs/references/opencode/README.md`
4. `docs/references/README.md`
5. `docs/references/AGENTS.md`

其中：

- 三个来源目录各保留一个主文件，符合“先以单文件为主”的决定。
- 根 README 负责阅读顺序、适用场景、后续 Phase 3/4 查阅入口。
- `docs/references/AGENTS.md` 负责把该目录的使用方式写清楚，和根 `AGENTS.md` 中现有层级声明对齐。

### Recommended Plan Split

- **Plan 01（Wave 1）**：完成 AstrBot 与 ACP 两份主参考文档。
- **Plan 02（Wave 2）**：完成 OpenCode 参考文档，并补齐根索引与 `docs/references/AGENTS.md`。

这样拆分的原因：

- AstrBot 与 ACP 可并行梳理，彼此无文件冲突。
- OpenCode 文档最好建立在 ACP 参考已成型的前提上，便于写“继承 ACP + OpenCode 差异/限制”。
- 根索引要最终指向三份已存在的文档，因此放在最后最稳。

## Risks / Pitfalls

- **不要把文档写成翻译全文。** 目标是“开发参考入口”，不是镜像站。
- **不要只写概念概览。** D-03 / D-04 / D-05 已锁定“需要做实”。
- **不要引入远程 ACP 方案。** 项目约束仍是本地 stdio。
- **不要把 OpenCode 文档写成通用产品使用手册。** 重点是 ACP server 规格与当前项目相关边界。
- **不要遗漏官方链接和限制说明。** 后续执行者需要知道“仓库摘要”和“原文规范基线”分别在哪里。

## Verification Guidance For Plans

计划中的自动验证应至少覆盖：

- 目标文件存在；
- 每份参考文档都包含“官方来源/原文链接”区块；
- AstrBot 文档覆盖插件入口、配置 schema、消息/会话、存储或 AI 能力；
- ACP 文档覆盖初始化、session、prompt turn、tool call / permission、content 或 schema 索引；
- OpenCode 文档覆盖 `opencode acp` 启动方式、stdio、支持范围、已知限制；
- 根 README 明确阅读顺序并链接到三份来源文档。

## Conclusion

这是 **Level 2 标准研究**：没有新运行时依赖，但需要基于多个官方来源做结构化整理。最合适的交付不是代码实现计划，而是 **两份可执行文档计划**，先沉淀 AstrBot/ACP 基线，再收口 OpenCode 与根索引。

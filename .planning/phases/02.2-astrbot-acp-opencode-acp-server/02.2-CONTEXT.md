<decisions>
## Implementation Decisions

### 文档定位

- **D-01:** `docs/references/` 的主产物定位为开发参考手册，而不是外部资料摘录。
- **D-02:** 文档需要服务后续 Phase 3/4 的研究与规划，重点是"开发时应该看什么、看到什么粒度、哪些内容算规范基线"。

### 内容范围

- **D-03:** AstrBot 插件开发文档需要做实，不只保留概念介绍。
- **D-04:** ACP 协议文档需要做实，不只保留概览。
- **D-05:** OpenCode ACP Server 也纳入本 phase，按完整规格整理处理，而不是只写简短互操作说明。

### 目录结构

- **D-06:** `docs/references/` 按来源分层组织（`astrbot/`、`acp/`、`opencode/`）。
- **D-07:** 每个来源目录先以单文件为主，避免一开始拆得过碎。
- **D-08:** `docs/references/README.md` 作为统一索引，说明三类文档的用途与阅读顺序。

### 资料采集规则

- **D-09:** 执行阶段以用户给定 URL 为起点，在同站点内尽量补齐相关信息。
- **D-10:** GitHub 来源优先继续查看仓库内 README、docs、spec 及相关讨论中的规范性内容。

### 规范基线

- **D-11:** 官方原文优先，作为真正的规范基线。
- **D-12:** 仓库内整理稿负责摘要、翻译、结构化重组，不替代官方原文。

### 实现映射

- **D-13:** 本 phase 先不强制要求文档显式标注"外部规范 ↔ 本仓库代码锚点"的映射关系。

### the agent's Discretion

- 具体文件命名
- 根索引中的阅读顺序文案
- 每份参考文档内部的小节切分方式
- 是否在不增加碎片化的前提下补少量附录或链接索引

</decisions>

<code_context>

## Existing Code Insights

### Reusable Assets

- `main.py`: 当前 AstrBot 插件注册、命令入口、权限确认、会话编排都集中在这里，后续参考文档应至少能支撑这些宿主侧能力的理解。
- `core/acp_client.py`: 已有通用 ACP JSON-RPC client，可作为研究 ACP initialize / session 系列方法时的实现参照。
- `core/acp_adapter.py`: 已有 OpenCode ACP payload 到插件内部模型的归一化层，可作为研究 OpenCode ACP Server 响应结构时的参照。
- `core/acp_models.py`: 已定义插件内部 ACP 数据模型边界，可辅助判断协议文档需要覆盖到哪些对象和字段。

### Established Patterns

- 项目已经明确采用本地 stdio ACP 链路，不扩展远程传输层。
- 当前仓库实现已围绕 ACP session、permission、mode/config option、event normalization 组织。
- 文档应优先服务现有架构，而不是引入新的抽象方向。

### Integration Points

- `docs/references/` 将成为后续 researcher / planner / executor 的参考入口。
- 后续 Phase 3/4 会直接依赖这些文档来理解宿主能力、协议约束和 OpenCode 行为边界。

</code_context>

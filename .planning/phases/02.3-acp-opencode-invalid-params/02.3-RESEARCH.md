## Summary

当前报错的第一根因已经可以高置信确认：插件发给 `opencode acp` 的 `initialize` 请求不是 ACP v1 形状。仓库现在发送的是 `params.capabilities`，并且缺少 `protocolVersion` 与 `clientInfo.version`；OpenCode 1.3.5 会立即返回 `-32602 Invalid params`。我已在本机直接用 `opencode acp` 复现：当前仓库形状必现同样错误，而按 ACP 官方 schema 改成 `protocolVersion + clientCapabilities + clientInfo.version` 后，请求立即成功。

但这不是唯一不兼容点。修完 initialize 后，当前实现还会继续撞上至少三处协议漂移：`session/new` 缺少必填 `mcpServers: []`；`session/prompt` 发送了非协议字段 `contentBlocks` 而不是 `prompt`；协议能力与 session/update 事件的解析也仍按旧字段读，导致后续会话恢复、mode 展示、进度/权限更新都不能正确工作。

**Primary recommendation:** 把这次修复当成一次“ACP v1 协议对齐”而不是单点热修：先统一 request/response shape，再补真实 OpenCode 交互测试，避免修完 initialize 后马上卡在下一跳。

## Architecture Patterns

### Recommended Project Structure

```text
core/
├── acp_client.py         # JSON-RPC method names + request payload 形状
├── executor.py           # session/new, load, prompt, capability 消费
├── acp_adapter.py        # OpenCode/ACP response 与 update 归一化
├── input.py              # AstrBot 输入 -> ACP ContentBlock
└── output.py             # ACP update/result -> 聊天消息
```

### Pattern 1: 在 `ACPClient` 层固定协议 method params

**What:** 所有 ACP method 的 request shape 在 client/executor 层一次定准，不让 `main.py` 直接拼协议字段。
**When to use:** 修 initialize / session/new / session/prompt / permission response 时。
**Example:**

```json
// Source: https://agentclientprotocol.com/protocol/initialization
{
  "method": "initialize",
  "params": {
    "protocolVersion": 1,
    "clientCapabilities": {
      "fs": { "readTextFile": true, "writeTextFile": true },
      "terminal": true
    },
    "clientInfo": {
      "name": "astrbot_plugin_acp",
      "title": "ACP Client",
      "version": "1.3.1"
    }
  }
}
```

### Pattern 2: 输入层产出“协议 prompt”，不是仓库私有 `contentBlocks`

**What:** `InputProcessor` 可以继续产出富内容，但最终要转成 ACP `prompt: ContentBlock[]`。
**When to use:** 发送文本、图片、附件、引用消息时。
**Example:**

```json
// Source: https://agentclientprotocol.com/protocol/prompt-turn
{
  "method": "session/prompt",
  "params": {
    "sessionId": "sess_xxx",
    "prompt": [{ "type": "text", "text": "请分析这个仓库" }]
  }
}
```

### Pattern 3: 统一在 adapter/output 层解包 `session/update`

**What:** OpenCode 的通知是 `params.sessionId + params.update`；内部显示层应先解包 `update.sessionUpdate`，再映射为聊天事件。
**When to use:** plan、tool_call、permission、mode/config 更新、available commands 更新。
**Example:**

```json
// Source: https://agentclientprotocol.com/protocol/prompt-turn
{
  "method": "session/update",
  "params": {
    "sessionId": "sess_xxx",
    "update": {
      "sessionUpdate": "tool_call",
      "toolCallId": "call_001",
      "title": "Reading configuration file",
      "status": "pending"
    }
  }
}
```

### Anti-Patterns to Avoid

- **把 ACP 字段名写死在多个层里：** 会导致 initialize 修好了、prompt 还错着。应集中到 client/executor/adapter。
- **继续把 `agentCapabilities` 当 `capabilities` 读：** 会让 `loadSession`、prompt/image 能力判断全部失真。
- **不传 `mcpServers`：** OpenCode 1.3.5 对 `session/new`/`session/load` 会直接报 `Invalid params`。
- **把 `modes` 当 list 读：** OpenCode 返回的是 `{ availableModes, currentModeId }` 对象，不是旧的裸数组。
- **不解包 `params.update`：** 会让 `session/update` 事件完全落空，权限确认和过程提示都收不到。

## Don't Hand-Roll

| Problem              | Don't Build                                                  | Use Instead                                             | Why                                                                |
| -------------------- | ------------------------------------------------------------ | ------------------------------------------------------- | ------------------------------------------------------------------ |
| ACP request 字段猜测 | 自己凭旧记忆拼 `capabilities` / `contentBlocks` / `configId` | ACP 官方 schema + 本机 `opencode acp` 实测              | 协议很严格，字段一错就是 `-32602 Invalid params`                   |
| OpenCode 支持面判断  | 靠 README 猜 method 是否可用                                 | `initialize.agentCapabilities` + `session/new` 实测响应 | `loadSession`、`list`、prompt 能力都应以运行时协商结果为准         |
| 会话事件解析         | 在 UI 层用 if/else 猜原始 payload                            | `acp_adapter.py` 做统一归一化                           | 事件包裹层 (`params.update`) 与不同 update 类型差异很大            |
| 媒体 prompt 表达     | 自造 `{type: "resource", uri: ...}` 简化版                   | ACP ContentBlock 正式结构                               | `resource` 与 `resource_link`、`image` 的 schema不同，后续极易踩坑 |

**Key insight:** 这类问题不是“OpenCode 启动失败”，而是“进程已启动，但第一批 JSON-RPC params 不满足 ACP schema”。所以不要再从进程管理方向手写兜底；应先把协议层对齐。

## Common Pitfalls

### Pitfall 1: `initialize` 沿用旧字段名

**What goes wrong:** OpenCode 立即返回 `Invalid params`，表面像“后端启动失败”。
**Why it happens:** 当前代码发送 `capabilities`，缺少 `protocolVersion` 与 `clientInfo.version`。
**How to avoid:** 改为 `protocolVersion`、`clientCapabilities`、完整 `clientInfo`。
**Warning signs:** stderr 中出现 `_errors.version`、`_errors.protocolVersion`、`expected null, received object`。

### Pitfall 2: 修完 initialize 后马上卡在 `session/new`

**What goes wrong:** 新会话创建继续报 `Invalid params`。
**Why it happens:** ACP v1 的 `session/new`/`session/load` 要求 `mcpServers` 必填；当前仓库没传。
**How to avoid:** 即使当前没有 MCP server，也显式传 `mcpServers: []`。
**Warning signs:** 返回错误里明确指向 `mcpServers` 缺失。

### Pitfall 3: prompt 仍发 `contentBlocks`

**What goes wrong:** 会话创建成功，但首次 `/oc` 提示继续报 `Invalid params`。
**Why it happens:** ACP `session/prompt` 需要 `prompt: ContentBlock[]`；当前仓库发的是 `contentBlocks`。
**How to avoid:** 在 `InputProcessor.to_payload()` 或 executor coercion 处统一产出 `prompt`。
**Warning signs:** 错误 data 指向 `prompt expected array, received undefined`。

### Pitfall 4: 能力与事件解析仍按旧响应结构

**What goes wrong:** `loadSession`、图片能力、mode 展示、进度消息、权限确认都异常或缺失。
**Why it happens:** 当前代码读取 `response.capabilities`、`payload.modes`、事件顶层 `type`，但 OpenCode 实际返回 `agentCapabilities`、`modes.availableModes/currentModeId`、`params.update.sessionUpdate`。
**How to avoid:** 在 adapter 中统一做 schema 对齐，内部再消费归一化结果。
**Warning signs:** initialize 成功但 `get_protocol_capabilities()` 为空；mode 一直显示“未提供”；运行时有 `session/update` 但聊天无过程消息。

### Pitfall 5: 忽略认证能力

**What goes wrong:** 即使协议修对，也可能在实际 prompt 时遭遇 `auth_required` 或需要 `authenticate` 流程。
**Why it happens:** OpenCode initialize 已返回 `authMethods`，当前仓库完全未实现该分支。
**How to avoid:** 本 phase 至少把它记录为已知后续分支；如果实测触发，再单独补认证流程，不要和参数修复混写。
**Warning signs:** initialize 响应有 `authMethods`；后续 `session/new`/prompt 返回认证相关错误。

### Pitfall 6: 本地 `stdio` 后端仍使用过长启动超时

**What goes wrong:** 参数错误或二进制错误本来会立即失败，但用户侧感知成“后端长时间启动失败”。
**Why it happens:** 当前默认 `acp_startup_timeout` 是 30 秒，这更像远端服务超时，不像本地子进程握手超时。
**How to avoid:** 对本地 `opencode acp` 默认使用短超时，建议 5 秒；需要时允许管理员手动调大。
**Warning signs:** `Invalid params` 这类即时错误被包装成统一的启动失败，且要等很久才返回。

## Open Questions

1. **OpenCode 是否要求先 authenticate 才允许真实 prompt?**
   - What we know: initialize 响应带 `authMethods`，当前仓库未处理。
   - What's unclear: 当前环境下 `session/new` 和真正 prompt 是否会触发 `auth_required`。
   - Recommendation: 在修完参数后做一次真实 `/oc hello` 冒烟；若触发认证，再拆成后续小 phase，不要阻塞这次协议对齐。

2. **图片/附件 block 是否需要从 `uri` 升级为 ACP 更标准的 `resource_link` / `resource` / image data?**
   - What we know: 协议文档对 `resource`、`resource_link`、`image` 的 shape 比当前仓库更严格。
   - What's unclear: OpenCode 对当前 `uri` 简化写法的兼容程度。
   - Recommendation: 本 phase 先修文本 prompt 主链路；媒体兼容单独补带 fixture 的测试。

3. **`session/set_config_option` 当前参数名是否要从 `optionId` 改成 `configId`?**
   - What we know: ACP schema 页面写的是 `configId`；当前仓库发 `optionId`。
   - What's unclear: OpenCode 1.3.5 是否兼容旧别名。
   - Recommendation: 顺手在本 phase 用本机 probe 验证；若不兼容，和 mode/config 解析一并修。

4. **`acp_startup_timeout` 默认值是否应从 30 秒收紧?**
   - What we know: 当前链路是本地 `stdio` 子进程，不是远端网络服务；本机错误返回非常快。
   - What's unclear: 极少数冷启动场景下是否会稳定超过 5 秒。
   - Recommendation: 本 phase 把默认值改为 5 秒，并保留配置项可调；验证时记录真实耗时。
